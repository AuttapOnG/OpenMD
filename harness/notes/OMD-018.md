# OMD-018 — Code block theme mismatch (hljs css vs in-page toggle) + stranded copy button

## Symptom (human screenshots, 2026-07-13)
1. OS light + page toggled dark → code block rendered as a white box on the
   dark page, with a dark copy button floating on it.
2. Even with matching schemes, the copy button sat visually *outside* the
   code box, in the pre's padding gutter.

## Root cause (two layers)
- hljs stylesheets were loaded with `media="(prefers-color-scheme: …)"`,
  i.e. they followed the OS, while the page theme follows the `data-theme`
  attribute set by the toggle. Mismatch whenever toggle ≠ OS.
- `hljs.highlightAll()` adds class `hljs` to the `<code>` element and the
  github themes set `.hljs { background; padding }`. `.hljs` (0,1,0) beats
  `pre code { background: transparent }` (0,0,2), so the code element drew
  its own inner box; the copy button is absolutely positioned in the `pre`
  (outer box), hence "stranded".

## Fix (src/render.ts)
- CSS: `pre code.hljs { background: transparent; padding: 0; }` — the pre
  is the single visible box again.
- Links get ids `hljs-css-light` / `hljs-css-dark`, no media query in
  markup (light starts `media="all"`, dark `media="not all"`).
- `setTheme()` computes the effective theme (auto → matchMedia) and flips
  the links' `media` between `all` / `not all`; a matchMedia change
  listener re-applies while in auto mode.

## Gotcha discovered during verification
Toggling `link.disabled` (first attempt) permanently detached the sheet in
Chromium — after disable→enable, `link.sheet` stayed null and syntax colors
were lost until reload. The `media` attribute swap is the reliable pattern.
