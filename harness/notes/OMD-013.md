# OMD-013 — Windows compatibility audit + CI test matrix

## Audit findings

| Site | Risk | Verdict + note |
|---|---|---|
| `extension.ts` `computeMirrorHtmlPath` (old) | drive letters, `HOME` vs `USERPROFILE`, case-sensitivity | **Fixed.** Extracted to `src/paths.ts`; the caller supplies `HOME || USERPROFILE || ''`. Injected `path.win32` tests cover drive-root paths, drive-letter removal, case-insensitive home comparison, marker matching, and an empty home directory. |
| `extension.ts` `urlPath` construction | backslashes/colons in URL | **Fixed.** `toUrlPath` converts native mirror separators to `/`; mirror-path construction filters drive-letter segments. Win32 tests assert that URLs contain neither backslashes nor colons. |
| `extension.ts` activation cleanup + asset staging | `path.join`/`rmSync`/`cpSync` semantics on Windows | **Already-safe.** All filesystem paths are formed with native `path.join`; recursive `rmSync`, `mkdirSync`, `copyFileSync`, and `cpSync` accept Windows paths. KaTeX's directory layout is preserved by recursive copy. |
| `extension.ts` webview `localResourceRoots` | `Uri.file`/`joinPath` on win32 | **Already-safe.** Source directories are converted with `vscode.Uri.file`, and extension media is constructed with `vscode.Uri.joinPath`; both APIs are platform-aware and avoid manual URI separator handling. |
| `server.ts` `/assets` handler | URL `/` split vs native join; traversal | **Already-safe.** It validates the decoded URL-relative asset name against a fixed whitelist or anchored KaTeX regex before splitting on URL `/` and joining with native `path.join`. Traversal and extra segments fail validation. |
| `server.ts` `/mtime` + page registry | registered key vs decoded request path (spaces/unicode) | **Already-safe.** Page requests are `decodeURIComponent`-decoded before lookup. The `/mtime` key is emitted with `encodeURIComponent`; Node's query parser decodes it before the same map lookup, so spaces and Unicode resolve to the registered decoded key. |
| `render.ts` | any fs/path use at all | **Justified.** There is no filesystem or path-module use in `render.ts`; it renders supplied Markdown and supplied asset URLs only, so it has no OS path dependency. |
| `src/test/**` | hardcoded POSIX fixture paths that would break on `windows-latest` | **Already-safe.** Existing filesystem fixtures use `os.tmpdir()` and `path.join`/`path.resolve`. The new literal POSIX and Windows paths are semantic inputs paired with explicitly injected `path.posix`/`path.win32`, not host filesystem paths. Integration tests contain no hardcoded filesystem paths. |
| `extension.ts` cleanup entry prefix matching (new finding) | `entry.startsWith(extensionId)` can include a different extension whose ID merely shares this prefix | **out-of-scope: needs new feature entry.** This is pre-existing cleanup-selection behavior rather than a Windows compatibility defect; cleanup should eventually require the extension ID followed by a version delimiter and validate the versioned directory form. |

## Verification status

- Codex compile and prescribed non-server unit commands pass.
- Host-only full server suite, package inspection, and real `windows-latest` CI remain pending.
